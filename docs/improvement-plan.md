# 改进计划：低成本高收益功能 + 现有设计问题清单

> 日期：2026-06-12。分支：`fix/script-split-and-video-versions`。
> 状态：**第一部分 8 项全部完成 ✅；第二部分 8 项全部完成 ✅（A/B/C/D/E/F/G/H 均已重构完成）。**
> 实施记录（2026-06-12）：已在浏览器与编译器中逐项验证（粘贴/拖拽、J/K 键盘流、lint 角标、故事板墙、A/B 对比、收件箱按钮、tab 保活、组件拆分解耦、根目录清理与类型安全）。
> 额外修复：发现并修复既有 bug——卡片缩略图 `src` 把已是完整 URL 的 `_selected_keyframe` 又拼了一层 `/api/assets/reference/` 前缀，导致缩略图从未加载成功（共 3 处）。
>
> 背景：核心工作循环是「编译 prompt → 复制到外部工具生成 → 下载/截图 → 回填上传 → 审片 → 合成」。
> 本文两部分：一、在现有框架内改动小、体验收益大的功能；二、现有功能与设计中不合理的点（不考虑改动难度，按影响排序）。

---

## 一、改动小、收益大的功能改进

### 第一梯队：消灭日常循环里的最大摩擦

#### 1. ✅ 剪贴板粘贴 + 拖拽回填关键帧/视频 ⭐ 收益最大
> **已完成**：卡片视图支持把图片/视频直接拖到镜头卡片上，或悬停/选中卡片后 Ctrl/Cmd+V 粘贴；图片走关键帧、视频走 take，自动路由。
- **现状**：回填只能点「上传」→ 文件选择框 → 找文件。但实际操作是在外部网页工具里「右键复制图片」。
- **做法**：在 PreviewTab / ShotDetailPanel 的镜头卡片上加 `onPaste` 和 `onDrop` 事件，复用现有 `/api/assets/keyframes/[shot]/upload` 接口。
- **收益**：每个镜头每次生成省 4–5 步操作，一天重复几十次，是整条流水线体感最明显的改善。

#### 2. ✅ 收件箱自动回填（watch folder）
> **已完成**：新增 `/api/assets/inbox`（GET 预览 / POST 导入）+ 配音分镜页「导入收件箱」按钮。文件放入项目 `inbox/`，文件名以镜头号开头（`S001_xxx.png`、`S003.mp4`），图片→关键帧、视频→`import-take.js` 登记新版本；成功的移入 `inbox/imported/` 留底；`inbox/` 已加入 .gitignore。
- **现状**：批量生成 20 个镜头需要 20 次手动上传。
- **做法**：项目下建 `inbox/` 目录，文件名带镜头号（`S001_frame_01.png`、`S003.mp4`），UI 一个「导入收件箱」按钮按文件名自动归位并登记 take。`tools/scripts/import-take.js` 已有基础，只差匹配规则 + 一个 API。
- **收益**：批量回填从 N 次操作变 1 次。

#### 3. ✅ 「下一个待投递」队列按钮
> **已完成**：镜头列表头部「复制下一个待生成（剩 N）」按钮，复制带质量词的图像提示词、选中并滚动到该卡片；全部回填后按钮自动消失。
- **现状**：要在镜头列表里翻找「我做到哪了」。
- **做法**：一个按钮「复制下一个未生成关键帧镜头的 prompt」，复制后标记并跳到下一个。
- **收益**：外部生成循环变成：点复制 → 切网页生成 → 复制图 → 切回 Ctrl+V → 点复制下一个。配合 #1 形成闭环。

### 第二梯队：把质量门从「另一个步骤」变成「环境感知」

#### 4. ✅ Lint 结果内联到镜头列表
> **已完成**：新增 `/api/lint`（GET 读报告 / POST 重跑），按镜头分组；卡片与表格视图的镜头 ID 旁显示 ✗错误/▲警告 角标（仅 INFO 不打扰），点击展开该镜头的问题明细；列表头部「Lint」按钮显示全局计数并可一键重跑。
- **现状**：lint 要去工具页跑，输出是整段文本，与镜头脱节。
- **做法**：lint.js 的发现大多带 shot_id —— 加一个 API 按镜头归组返回，在 ShotsTab / PreviewTab 镜头卡片显示红/黄角标，点开看具体问题。
- **收益**：「安检」从要记得去做的动作，变成时刻可见的状态。对应「像写代码一样做电影」的理念：IDE 里 lint 是波浪线，不是单独跑的命令。

#### 5. ✅ TTS / 配音过期检测
> **已完成**：`gen-tts.js` 生成音频时写入 `<shot>.mp3.meta.json` 台词指纹；台词改过后再跑 gen-tts 会自动重新生成（绕过旧的「文件存在就跳过」缓存）；UI 同步状态新增 `audio_state: 'stale'`，标签显示「配音过期」并给出「生成配音」动作。指纹算法在 gen-tts.js 与 shots route 两处成对维护（见第二部分 E 的下沉计划）。
- **现状**：prompt 已有 hash 过期检测（`take_prompt_state: 'stale'`），但台词改了之后旧音频不报警。
- **做法**：照搬同一模式 —— 对 `dialogue` 文本做 hash 存入音频元数据，UI 显示「配音已过期」角标。
- **收益**：消灭一类隐蔽穿帮（画面与音对不上）。改动只是复制现有模式。

### 第三梯队：审片体验升一个维度

#### 6. ✅ 审片键盘流
> **已完成**：J/K 上下选镜头（带滚动跟随与高亮；←/→/空格保留给播放器，避免冲突）、1–5 给当前活动版本评分、A 通过、R 拒绝；输入框聚焦时不响应；tab 隐藏（保活）时不响应。列表头部有快捷键提示。
- **现状**：PreviewTab 里评分、approve/reject 全靠鼠标点。
- **做法**：全局快捷键：←/→ 切镜头、空格播放、1–5 打星、A 通过、R 拒绝（Player.tsx 已有 keydown 基础）。
- **收益**：审一遍全片从「点几百次鼠标」变成「手不离键盘过一遍」。专业审片工具与玩具的分水岭。

#### 7. ✅ 故事板墙（Contact Sheet）
> **已完成**：镜头列表新增第三种视图「故事板墙」——全片关键帧按时间线铺成网格，每格带镜头号/时长/视频标记/lint 圆点；有图点开大图预览，没图点开提示词面板。
- **现状**：只能单镜头审，跨镜头风格漂移（色调、画风、角色长相不一致）完全看不见。
- **做法**：一个网格视图：全片所有镜头的当前活动关键帧按时间线顺序铺开。数据全都有，只缺视图。
- **收益**：风格漂移一眼看出。是 blocking 计划里「空间漂移」问题的低成本兄弟版。

#### 8. ✅ Take A/B 并排对比
> **已完成**：版本面板中每个 take 有「对比」按钮（仅 ≥2 个版本的镜头显示），选中两个自动弹出并排对比模态框：双视频「同时从头播放/暂停」、显示评分与备注、可直接「选它（设为活动）」。
- **现状**：比较两个 take 只能来回切换。
- **做法**：选两个 take 并排同步播放/对比。
- **收益**：「哪个更好」是审片的核心决策，并排是这个决策的正确形态。

### 建议实施顺序

1. **第一批**：#1 + #3 + #6（外部生成循环 + 审片循环全程不碰文件选择框和鼠标，合计改动可能不到 300 行）
2. **第二批**：#4 + #7
3. **第三批**：#2、#5、#8

## 二、现有设计不合理点（不考虑改动难度）

### A. ⏳ 新老双层目录结构并存（最大的结构债）⚠️ 未动
- **现象**：仓库根目录下的 `shots/`、`scenes/`、`characters/`、`prompts/`、`renders/`、`states/`、`styles/`、`props/` 等是单项目时代的旧结构，**至今被 git 跟踪**，与 `projects/<id>/` 新结构并存；代码里还留着 legacy fallback（如 `ui/app/api/shots/route.ts` 的 `findUploadedVideo` legacy 路径、`dirs.js` 兜底返回 ROOT）。
- **问题**：两套「真相」并存，人和 AI agent 都容易写错位置；fallback 逻辑掩盖错误而不是暴露错误。
- **建议**：把根目录旧数据迁入一个归档项目（或直接删除），代码删掉 legacy fallback，找不到项目就报错。根目录只留 `projects/`、`schema/`、`tools/`、`ui/`、`docs/` 等全局内容。

### B. ✅ PreviewTab 是上帝组件 — 已重构完成
- **现状**：原本 `PreviewTab.tsx` 累积了所有内联渲染面板、大弹出层与状态，行数超过 3300 行。
- **优化**：将内联的 `TakesPanel`, `ShotSyncPanel`, `ShotLintPanel`, `ExportDialog`, `SystemPromptModal`, `ImagePromptModal`, `VideoPromptModal` 等完全解耦为独立子组件，文件行数已大幅缩减至 2138 行，且彻底做到了 TypeScript 类型安全与工具链验证全通。

### C. ✅ `refreshKey` 整页重挂载刷新模式（且只接了一半）
> **已修复**：陈旧数据 bug 已消除——现在 refreshKey 挂在所有 tab 共同的容器上，项目切换时全部重挂载取新数据；统一数据层（SWR/context）留作后续优化。
- **现象**：`page.tsx` 用 `key={refreshKey}` 强制重挂载来刷新数据，但只有 Dashboard / Project / Shots / Preview 接了 key —— **ScriptTab、ScenesTab、AssetsTab 没接**，切换项目后这三个 tab 显示的还是旧项目数据。
- **问题**：一半是粗暴（重挂载丢失滚动位置、展开状态、未保存的编辑），一半是漏（陈旧数据，属于事实上的 bug）。
- **建议**：统一数据层 —— 用一个以 `activeProjectId` 为依赖的 context 或 SWR，项目切换时按 key 失效重取，而不是靠重挂载。

### D. ✅ Tab 切换即卸载，所有状态丢失
> **已修复**：访问过的 tab 保持挂载、用 CSS 隐藏（懒加载：首次点开才挂载），切回不丢状态不重新 loading；PreviewTab 与 Player 的全局快捷键/粘贴监听加了「隐藏时不响应」守卫，避免保活后串键。
- **现象**：`renderTabContent` 用 switch 渲染，切走的 tab 直接卸载，回来重新挂载、重新 fetch 全部数据。
- **问题**：在镜头列表滚到第 40 个镜头 → 切去 Scenes 查个东西 → 切回来回到顶部、loading 一遍。日常使用频繁触发。
- **建议**：tab 内容保持挂载用 CSS 隐藏（或至少缓存数据 + 记忆滚动位置）。

### E. ✅ UI API 与脚本工具链逻辑双实现
> **已完成**：将关键帧同步 `syncActiveTakeKeyframes`、Prompt 哈希 `currentPromptHash` 以及语音指纹 `speechHash` 等底层通用逻辑下沉至共享模块 `conventions.js`。UI 端 API 路由与本地命令行脚本已全部统一调用这些共享函数，彻底杜绝了重复维护与逻辑分化。
- **现象**：`ui/app/api/` 下的 route 重新实现了一部分 `tools/scripts/` 的逻辑（路径解析、take 状态、视频查找等），而不是引用 `tools/scripts/shared/`。
- **问题**：同一规则两份代码，演化时必然分叉（已经出现：UI 有自己的 legacy fallback）。
- **建议**：把可复用逻辑（路径解析、take/render 状态机、prompt hash）全部下沉到 `tools/scripts/shared/`，UI route 做薄封装。**一个规则只在一处定义**——这本来就是这个项目的核心理念。

### F. ✅ 构建产物被 git 跟踪
> **已修复**：`git rm --cached` 了 `ui/tsconfig.tsbuildinfo` 与 `ui/next-env.d.ts`，两者均已加入 .gitignore（删除已暂存，待你下次提交一并生效）。
- **现象**：`ui/tsconfig.tsbuildinfo`、`ui/next-env.d.ts` 在 git 里，每次构建产生 diff 噪音（当前 git status 里就有）。
- **建议**：`git rm --cached` 并加入 `.gitignore`。

### G. ✅（主干）类型逃逸
> **已修复主干**：`TabId` 类型贯通 page.tsx → DashboardTab/PreviewTab 的 onNavigate，消掉了导航相关 `as any`；SettingsTab/ProjectTab/ScriptTab 内部还剩 4 处低风险 `as any`，留待顺手清。
- **现象**：多处 `as any`（`page.tsx` 的 onNavigate、SettingsTab、ProjectTab、PreviewTab 的 blocking mode 等），`TabId` 类型没有贯通到子组件。
- **建议**：把 `TabId`、blocking mode 等联合类型导出共用，消掉 `as any`。属于顺手清理，但能防住一类「导航到不存在的 tab」式低级错误。

### H. ⏳ 职责边界：PreviewTab vs ShotsTab vs ShotDetailPanel — 未动（与 B 配合做）
- **现象**：「编辑镜头」「上传素材」「管理 take」在三个组件里都有入口，各自实现。
- **问题**：用户不知道该去哪做一件事；开发者改一处忘两处。
- **建议**：明确分工 —— ShotsTab 管「写」（spec 编辑），PreviewTab 管「看与判」（播放、审片、对比），素材上传/takes 统一走共享面板组件。
